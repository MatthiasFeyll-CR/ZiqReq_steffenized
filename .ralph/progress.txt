## BF-001: Fix: DEF-002 — Replace serialized_rollback with app-table TRUNCATE in conftest.py
- Status: PASS
- Attempts: 4
- Changes: services/gateway/conftest.py
- Tests: 142 Python tests passed (0 errors, 0 failures), 244 Node tests passed, ruff lint clean, tsc clean
- Learnings for future iterations:
  - Django 5.2's `create_contenttypes` and `create_permissions` post_migrate handlers use `bulk_create` WITHOUT `ignore_conflicts` on PostgreSQL, causing duplicate-key IntegrityErrors when content_type/permission rows already exist (e.g., with keepdb=True)
  - `create_permissions` is registered with `dispatch_uid="django.contrib.auth.management.create_permissions"` — must pass this uid when calling `post_migrate.disconnect()`, otherwise the disconnect silently does nothing
  - `create_contenttypes` is registered WITHOUT a dispatch_uid, so plain `post_migrate.disconnect(create_contenttypes)` works
  - The fix: disconnect both handlers BEFORE `setup_databases()` and keep them disconnected for the entire test session, so both initial migration and TransactionTestCase flush teardowns skip the problematic bulk_create calls
  - Initial TRUNCATE-based approach failed because: (1) some table names in _APP_TABLES didn't exist in the DB, (2) Django's TransactionTestCase._fixture_teardown still calls flush → emit_post_migrate_signal independently
  - Session-scoped autouse fixtures don't have guaranteed ordering relative to other session-scoped fixtures (like django_db_setup) — moving signal disconnection INTO django_db_setup was necessary
  - Transient test failures can occur when the DB has stale data from previous failed runs — always use `docker compose down -v` (with `-v` for volumes) before re-running

## BF-001: Fix: DEF-003 — Add DB connection cleanup fixture to conftest.py
- Status: PASS
- Attempts: 3
- Changes: services/gateway/conftest.py
- Tests: 142 Python tests passed (0 errors, 0 failures), 244 Node tests passed, ruff lint clean, tsc clean
- Learnings for future iterations:
  - Stale thread-local DB connections between async TransactionTestCase tests cause order-dependent failures where WebSocket connections are rejected or subscribe returns "Access denied"
  - `connections.close_all()` only closes connections for the CURRENT thread — it does NOT affect the ASGIRef main thread used by `@database_sync_to_async(thread_sensitive=True)`
  - An autouse fixture with `connections.close_all()` must guard against closing connections inside atomic blocks (regular `TestCase` wraps tests in transactions) — use `if not connection.in_atomic_block` check
  - Placing the fixture in the root conftest (with the atomic block guard) is necessary — a websocket-only conftest doesn't help because the stale connections come from non-WebSocket tests that run first
  - After closing all connections, also explicitly close `connection.connection` and set it to None to reset the default connection's thread-local state
  - First run after `docker compose down -v` may show transient failures from stale DB state; second run confirms stability
