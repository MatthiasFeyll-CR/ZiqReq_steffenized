- Learnings for future iterations:
  - WebSocketAuthMiddleware is an ASGI middleware wrapping URLRouter in asgi.py. It validates token from query string and populates scope["user_id"] and scope["user_display_name"]

  - Consumer rejects connection (close code 4003) if scope["user_id"] is missing (middleware didn't set it)
  - receive_json dispatches to handle_{type} methods, returns error JSON for unknown/missing types
  - Tests use channels.testing.WebsocketCommunicator with the full middleware+router stack
  - Docker test containers copy code at build time — must rebuild python-tests after code changes
  - InMemoryChannelLayer added to test settings for channel layer operations
  - WebsocketCommunicator does NOT expose .instance attribute — can't directly inspect consumer state in tests
  - The docker compose build gate check has a pre-existing failure (frontend Dockerfile references nginx.conf that doesn't exist at that path)

## US-002: Channel group management — subscribe_idea, unsubscribe_idea, group join/leave
- Status: PASS
- Attempts: 1
- Changes: services/gateway/apps/websocket/consumers.py, services/gateway/apps/websocket/tests/test_consumers.py, .ralph/test-manifest.json
- Tests: 16 consumer tests passed (6 from US-001 + 10 new). Full suite: 126 Python tests passed, 191 Node tests passed.
- Learnings for future iterations:
  - handle_subscribe_idea validates UUID format, checks access (owner/co_owner/collaborator via Idea + IdeaCollaborator models), then does group_add
  - handle_unsubscribe_idea is a simple group_discard + remove from subscribed_groups set
  - Channel group naming convention: "idea_{uuid}"
  - Access check uses database_sync_to_async wrapper to query gateway's Idea and IdeaCollaborator models
  - Successful subscribe/unsubscribe are silent (no response sent) — only errors send a response
  - Duplicate subscribe is a no-op (checks subscribed_groups set first)
  - Unsubscribe for non-subscribed group is also a no-op
  - Running tests on individual file fails with content_type UniqueViolation — must run via full pytest (conftest handles persistent test DB)

## US-003: Chat message broadcast — on REST POST success, broadcast chat.message.created to idea group
- Status: PASS
- Attempts: 1
- Changes: services/gateway/apps/websocket/consumers.py, services/gateway/apps/chat/views.py, services/gateway/apps/websocket/tests/test_consumers.py, .ralph/test-manifest.json
- Tests: 3 new tests passed (test_chat_message_broadcast_to_subscriber, test_chat_message_broadcast_not_received_by_unsubscribed, test_chat_message_broadcast_via_view_helper). Full suite: 128 Python tests passed, 191 Node tests passed. Ruff clean. TSC clean.
- Learnings for future iterations:
  - Broadcasting from a sync Django REST view uses `async_to_sync(channel_layer.group_send)()` pattern (from asgiref)
  - `get_channel_layer()` returns the configured channel layer (InMemoryChannelLayer in tests, Redis in prod)
  - Consumer group message handlers (e.g. `chat_message`) are called by channel layer's group_send — the `type` field maps to method name (dots become underscores)
  - The `_broadcast_chat_message` helper is a fire-and-forget with exception logging — broadcast failure doesn't break the REST response
  - Payload includes: id, sender_type, sender{id, display_name}, ai_agent, content, message_type, created_at — matching the WebSocket protocol spec
  - No RabbitMQ event publishing exists yet in the codebase — the "broker event" from the PRD is implemented directly via channel_layer.group_send instead

- Learnings for future iterations:

  - Consumer `board_update` handler is a simple passthrough like `chat_message` — forwards group_send to WebSocket client
  - All 6 board CRUD endpoints (create/update/delete for both nodes and connections) now broadcast after persistence
